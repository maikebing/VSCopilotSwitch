# Roadmap

本文档描述 VSCopilotSwitch 的阶段性路线图。路线图会随着协议验证、OmniHost 集成和 VS Code 配置行为变化持续调整。

本次路线图按 `AGENTS.md` 工作原则的最后两项重新排序：先处理 Windows、macOS、Linux、WSL 的路径、权限、托盘能力和后台进程差异；再让 UI 的关键写入动作可理解、可预览、可回滚，降低误操作风险。

目标：建立可运行的工程骨架和清晰模块边界。

- [x] 建立 README、路线图、智能体要求和变更日志。
- [x] 初始化 .NET 解决方案。
- [x] 建立 `src/VSCopilotSwitch` 宿主项目，最终可执行文件名为 `VSCopilotSwitch`。
- [x] 建立 `VSCopilotSwitch.Core`、`VSCopilotSwitch.VsCodeConfig`、`VSCopilotSwitch.Ui` 模块。
- [x] 初始化 Vue 3 SPA 工程文件和项目内 npm 脚本。
- [x] ✅️ 按 Visual Studio SPA 模式加入前端 `.esproj`，并让后端通过 `SpaProxy` 关联 Vue 调试服务。
- [x] ✅️ 将前端工程切换为 VueApp2 同款 Vue 3 + TypeScript 技术栈。

## 阶段 1：四端运行基线

目标：优先解决 Windows、macOS、Linux、WSL 的路径、权限、托盘能力和后台进程差异，避免后续功能在四端反复返工。

- [ ] 定义平台能力矩阵：窗口、托盘、后台进程、文件权限、开机启动、配置路径。
- [ ] 实现平台识别：Windows、macOS、Linux、WSL。
- [ ] 实现 VS Code User 配置目录发现的四端测试样例。
- [ ] 明确 WSL 下 Windows 侧 VS Code 配置和 Linux 侧配置的选择策略。
- [ ] 实现本地代理监听地址和端口占用检测。
- [ ] 默认只监听 `127.0.0.1`。
- [ ] 设计后台代理生命周期：启动、停止、崩溃恢复、退出清理。
- [ ] 设计托盘能力抽象，兼容不同平台托盘限制。
- [ ] 确认 OmniHost 窗口和托盘 API 的落地方式。

验收标准：

- 四端路径选择逻辑有明确测试或人工验收清单。
- WSL 场景不会静默修改用户未选择的配置文件。
- 退出程序时本地代理能安全停止。
- 托盘不可用的平台能降级到窗口内控制，不阻塞核心功能。

## 阶段 2：低误操作 UI 与配置向导

目标：所有关键写入动作都让用户能理解“将修改什么、为什么修改、如何恢复”。

- [x] 建立 Vue 3 + TypeScript SPA 主界面骨架。
- [x] ✅️ 展示当前代理状态、模型列表和供应商切换入口。
- [x] ✅️ 展示 VS Code 配置 dry-run 预览入口。
- [ ] 实现配置写入向导：选择 VS Code 配置目录、预览差异、确认写入、显示结果。
- [ ] 实现差异视图：展示 `settings.json` 和 `chatLanguageModels.json` 的字段级变化。
- [ ] 实现回滚入口：列出最近备份并恢复。
- [ ] 实现高风险动作确认：写入配置、包含密钥导出、退出并停止代理。
- [ ] 实现失败修复建议：权限不足、JSON 无效、文件被占用、端口冲突。
- [ ] 默认折叠高级选项：代理、熔断阈值、重试次数、备用路由。

验收标准：

- 用户在点击写入前能看到目标文件、修改字段和备份位置。
- 默认流程不会要求用户理解内部协议细节。
- 所有失败状态都有可执行的下一步建议。
- 重复执行配置向导不会产生重复条目。

## 阶段 6：稳定性与路由

目标：让多供应商聚合具备生产可用的容错能力，并在 UI 中清楚解释当前路由状态。

- [ ] 熔断器：失败阈值、半开探测、自动恢复。
- [ ] 重试策略：可按供应商、模型和错误类型配置。
- [ ] 超时控制：连接超时、首 token 超时、总请求超时。
- [ ] 备用路由：主模型失败后切换到备用模型。
- [ ] 限流策略：按供应商、模型和 API Key 控制并发。
- [ ] 健康检查：API Key、模型列表和聊天探测。
- [ ] 本地日志：脱敏记录请求摘要、耗时和失败原因。
- [ ] UI 展示熔断、重试、降级和健康状态，并避免用模糊状态误导用户。

验收标准：

- 单个供应商故障不会拖垮整个本地代理。
- 熔断状态在 UI 和托盘菜单中可见。
- 敏感信息不会出现在日志中。

## 关键风险

- VS Code / Copilot Chat 的配置格式可能变化，需要保持文档追踪和版本兼容。
- 各供应商的流式响应语义不同，工具调用、多模态、推理内容字段需要逐步兼容。
- WSL 场景路径复杂，需要避免误改用户不期望的配置文件。
- 中转站协议差异较大，需要通过可扩展 Adapter 处理站点私有字段。
- 本地代理如果处理不当，可能泄露 API Key 或请求内容，必须优先设计脱敏和权限边界。
- SPA 资源嵌入单体应用后，需要处理缓存、版本号和本地调试资源切换，避免开发资源泄露到正式包。
- AOT 和单文件发布可能影响反射、动态加载和资源访问方式，需要在早期验证。
- UI 如果把“预览”和“写入”混在一起，用户可能误改 VS Code 配置，必须拆成独立步骤。
