# Roadmap

本文档描述 VSCopilotSwitch 的阶段性路线图。路线图会随着协议验证、OmniHost 集成和 VS Code 配置行为变化持续调整。

## 阶段 0：项目基线

目标：建立清晰的产品、工程和协作边界。

- [x] 明确项目目标：把多供应商模型转换为 Ollama 兼容协议。
- [x] 建立 README、路线图、智能体要求和变更日志。
- [ ] 确认 OmniHost 作为四端界面宿主的初始化方式。
- [ ] 确认 Vue 3 SPA 的目录结构、npm 脚本、开发代理和构建输出目录。
- [ ] 定义项目目录结构和模块边界。
- [ ] 确认 AOT 单体应用的发布目标，以及 SPA 静态文件作为嵌入式资源的打包方式。
- [ ] 明确许可证、贡献方式和安全边界。

交付物：

- `README.md`
- `ROADMAP.md`
- `AGENTS.md`
- `CHANGELOG.md`
- 初始工程骨架

## 阶段 1：VS Code 配置管理 MVP

目标：安全、可回滚地修改 VS Code 的 Ollama 相关配置。

- [ ] 实现跨平台 VS Code User 配置目录发现。
- [ ] 支持 Windows、macOS、Linux、WSL 路径检测。
- [ ] 读取 `settings.json` 和 `chatLanguageModels.json`。
- [ ] 实现 JSON 容错读取、格式化写入和字段级更新。
- [ ] 写入前自动创建时间戳备份。
- [ ] 提供配置差异预览。
- [ ] 提供一键回滚到最近备份。
- [ ] 避免覆盖非本项目管理的用户配置。

验收标准：

- 重复执行不会产生重复配置。
- 用户已有设置不会被删除。
- 任意写入失败时可以恢复到写入前状态。
- WSL 场景下能明确提示当前将修改 Windows 侧还是 Linux 侧配置。

## 阶段 2：Ollama 兼容代理 MVP

目标：启动本地服务，把上游供应商模型暴露为 Ollama 风格接口。

- [ ] 实现 `/api/tags` 模型列表接口。
- [ ] 实现 `/api/chat` 非流式聊天接口。
- [ ] 实现 `/api/chat` 流式聊天接口。
- [ ] 实现基础错误映射。
- [ ] 支持模型别名和供应商路由。
- [ ] 支持本地端口配置和端口占用检测。

验收标准：

- VS Code 可通过 Ollama 相关配置发现并调用代理模型。
- 代理服务能返回稳定的 Ollama 兼容响应结构。
- 上游错误不会直接泄露敏感字段。

## 阶段 3：首批 Provider Adapter

目标：支持主流模型供应商和中转协议。

- [ ] sub2api 中转站协议。
- [ ] OpenAI Official。
- [ ] Claude Official。
- [ ] DeepSeek。
- [ ] NVIDIA NIM / build.nvidia.com。
- [ ] Moark。

每个 Adapter 需要包含：

- 模型列表获取。
- Chat 请求转换。
- 流式响应转换。
- 错误码映射。
- 鉴权配置。
- 最小集成测试。

## 阶段 4：稳定性与路由

目标：让多供应商聚合具备生产可用的容错能力。

- [ ] 熔断器：失败阈值、半开探测、自动恢复。
- [ ] 重试策略：可按供应商、模型和错误类型配置。
- [ ] 超时控制：连接超时、首 token 超时、总请求超时。
- [ ] 备用路由：主模型失败后切换到备用模型。
- [ ] 限流策略：按供应商、模型和 API Key 控制并发。
- [ ] 健康检查：API Key、模型列表和聊天探测。
- [ ] 本地日志：脱敏记录请求摘要、耗时和失败原因。

验收标准：

- 单个供应商故障不会拖垮整个本地代理。
- 熔断状态在 UI 中可见。
- 敏感信息不会出现在日志中。

## 关键风险

- VS Code / Copilot Chat 的配置格式可能变化，需要保持文档追踪和版本兼容。
- 各供应商的流式响应语义不同，工具调用、多模态、推理内容字段需要逐步兼容。
- WSL 场景路径复杂，需要避免误改用户不期望的配置文件。
- 中转站协议差异较大，需要通过可扩展 Adapter 处理站点私有字段。
- 本地代理如果处理不当，可能泄露 API Key 或请求内容，必须优先设计脱敏和权限边界。
- SPA 资源嵌入单体应用后，需要处理缓存、版本号和本地调试资源切换，避免开发资源泄露到正式包。
- AOT 和单文件发布可能影响反射、动态加载和资源访问方式，需要在早期验证。
